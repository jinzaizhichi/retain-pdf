# 任务详情与事件

这篇讲任务执行过程怎么读。

## 主接口

任务详情：

- `GET /api/v1/jobs/{job_id}`

事件流：

- `GET /api/v1/jobs/{job_id}/events`
- `GET /api/v1/ocr/jobs/{job_id}/events`

失败诊断与恢复：

- `GET /api/v1/jobs/{job_id}/diagnostics`
- `GET /api/v1/jobs/{job_id}/resume-plan`
- `POST /api/v1/jobs/{job_id}/resume`

## 任务详情里该看什么

重点字段：

- `status`
- `stage`
- `stage_detail`
- `runtime`
- `failure`
- `failure_diagnostic`
- `timestamps`
- `actions`
- `artifacts`
- `ocr_job`
- `invocation`
- `normalization_summary`
- `glossary_summary`
- `log_tail`

推荐前端用法：

- 当前阶段说明看 `stage_detail`
- 当前任务真阶段看 `runtime.current_stage`
- 总耗时看 `runtime.total_elapsed_ms`
- 当前阶段耗时看 `runtime.active_stage_elapsed_ms`
- 过程时间线看 `runtime.stage_history`

## 时间线真源

`runtime.stage_history` 是主时间线真源，不要从 `/events` 反推。

每一项通常包含：

- `stage`
- `detail`
- `enter_at`
- `exit_at`
- `duration_ms`
- `terminal_status`

## 事件流怎么读

事件流适合排障，不是主时间线。

事件项常见字段：

- `seq`
- `ts`
- `level`
- `user_stage`
- `stage`
- `substage`
- `stage_detail`
- `event`
- `event_type`
- `message`
- `progress_current`
- `progress_total`
- `progress_unit`
- `payload`

推荐优先看：

- `event_type`
- `user_stage`
- `stage`
- `stage_detail`
- `progress_current`
- `progress_total`
- `progress_unit`
- `message`
- `payload`

前端状态卡优先消费 `event_type=stage_progress` 的事件。后端会持续为这些阶段写进度：

- `continuation_review`：`progress_unit=page`
- `page_policies`：`progress_unit=page`
- `domain_inference`：`progress_unit=page`
- `garbled_repair`：`progress_unit=page`
- `rendering`：`progress_unit=page`

批量翻译阶段使用 `progress_unit=batch`。Typst 编译、保存等不可按页拆分的阶段使用 `progress_unit=step`。

## 生命周期

常见 workflow：

- `book`
- `ocr`
- `translate`
- `render`

常见主阶段：

- `queued`
- `ocr_submitting`
- `ocr_processing`
- `normalization`
- `translation_prepare`
- `translating`
- `rendering`
- `saving`
- `finished`
- `failed`

## 失败协议

失败时优先看：

- `failure.summary`
- `failure.category`
- `failure.root_cause`
- `failure.retryable`
- `failure.suggestion`

兼容字段：

- `failure_diagnostic`
- `log_tail`

前端如果只需要任务详情页的干净失败信息，优先读：

`GET /api/v1/jobs/{job_id}/diagnostics`

返回字段：

- `failed_stage`
- `failed_substage`
- `summary`
- `detail`
- `suggestion`
- `retryable`
- `resume_available`

这个接口会复用 `failure` 的正式字段，并在必要时回退到后端分类器，不要求前端自己解析 `request_payload`、`events` 或 `log_tail`。

## 断点恢复

恢复计划：

`GET /api/v1/jobs/{job_id}/resume-plan`

返回字段：

- `can_resume`
- `job_id`
- `from_stage`
- `resume_workflow`
- `reuses_artifacts`
- `reruns_stages`
- `reason`

当前规则：

- 有 `translations_dir + source_pdf`：`from_stage=render`，`resume_workflow=render`，只重跑渲染。
- 有 `normalized_document_json + source_pdf`：`from_stage=translate`，`resume_workflow=book`，重跑翻译和渲染。
- 两类 checkpoint 都没有：`can_resume=false`，`reason` 给出原因。

执行恢复：

`POST /api/v1/jobs/{job_id}/resume`

当前实现复用 `/rerun` 的执行契约：

- 可原地重渲染时，保留原 `job_id`，清空旧渲染产物，排队 `render` workflow。
- 只有 OCR checkpoint 时，创建新的 `book` 恢复任务。
- 如果任务仍在 `queued` / `running`，会返回状态冲突，需要先取消。

## 关键结论

- 详情页主接口是 `GET /api/v1/jobs/{job_id}`
- 时间线看 `runtime.stage_history`
- 事件流看 `GET /api/v1/jobs/{job_id}/events`
- 失败真源看 `failure`
- 简洁失败信息看 `GET /api/v1/jobs/{job_id}/diagnostics`
- 恢复按钮状态看 `GET /api/v1/jobs/{job_id}/resume-plan`
