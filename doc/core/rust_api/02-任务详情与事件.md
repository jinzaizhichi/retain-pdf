# 任务详情与事件

这篇讲任务执行过程怎么读。

## 主接口

任务详情：

- `GET /api/v1/jobs/{job_id}`

事件流：

- `GET /api/v1/jobs/{job_id}/events`
- `GET /api/v1/ocr/jobs/{job_id}/events`

失败诊断与恢复：

- `GET /api/v1/jobs/{job_id}/diagnostics`
- `GET /api/v1/jobs/{job_id}/resume-plan`
- `POST /api/v1/jobs/{job_id}/resume`

## 任务详情里该看什么

重点字段：

- `status`
- `stage`
- `stage_detail`
- `runtime`
- `failure`
- `failure_diagnostic`
- `timestamps`
- `actions`
- `artifacts`
- `ocr_job`
- `invocation`
- `normalization_summary`
- `glossary_summary`
- `log_tail`

推荐前端用法：

- 当前阶段说明看 `stage_detail`
- 当前任务真阶段看 `runtime.current_stage`
- 总耗时看 `runtime.total_elapsed_ms`
- 当前阶段耗时看 `runtime.active_stage_elapsed_ms`
- 过程时间线看 `runtime.stage_history`

## 时间线真源

`runtime.stage_history` 是主时间线真源，不要从 `/events` 反推。

每一项通常包含：

- `stage`
- `detail`
- `enter_at`
- `exit_at`
- `duration_ms`
- `terminal_status`

## 事件流怎么读

事件流适合状态卡、进度展示和排障，不是主时间线。主时间线仍看 `runtime.stage_history`。

`GET /api/v1/jobs/{job_id}/events` 返回的是 Rust canonicalize 后的公共事件，不是 Python 原始事件。公共事件里：

- `stage` 是给前端消费的公共阶段，只能理解为 `ocr | translation | render | done` 这一层。
- `substage` 是机器可读子阶段，例如 `translation_batches`、`render_prewarm`。
- `event_type` 是归一化事件类型，例如 `progress`、`artifact`、`terminal`、`error`、`diagnostic`。
- `message` 只给人看，前端不要靠它判断阶段。
- 原始 Python `stage` / `user_stage` / `event_type` 会保存在 `payload.raw_stage`、`payload.raw_user_stage`、`payload.raw_event_type` 中，排障时再看。

事件项稳定字段：

- `seq`
- `ts`
- `created_at`
- `level`
- `lane`
- `stage`
- `substage`
- `stage_detail`
- `event`
- `event_type`
- `raw_event_type`
- `progress`
- `message`
- `provider`
- `provider_stage`
- `payload`

`progress` 是公共进度对象：

```json
{
  "unit": "page",
  "current": 37,
  "total": 142,
  "percent": 26.056338028169012
}
```

前端状态卡优先消费：

- `stage`
- `substage`
- `stage_detail`
- `event_type`
- `progress.unit`
- `progress.current`
- `progress.total`
- `progress.percent`

不要优先消费这些内部/兼容字段：

- `payload.raw_stage`
- `payload.raw_user_stage`
- `payload.raw_event_type`
- `message`
- `provider_stage`

`lane` 用来区分主进度和后台事件：

- `main`：主状态卡可展示。
- `background`：后台预热、后台准备等，不应覆盖主阶段。
- `artifact`：产物发布。
- `diagnostic`：诊断或错误辅助信息。

当前稳定子阶段：

| 公共阶段 | 子阶段 | 进度单位 | 说明 |
| --- | --- | --- | --- |
| `ocr` | `ocr_processing` | `page` | OCR 解析页进度；如果 provider 只能返回完成态，则至少给总页数。 |
| `ocr` | `normalizing` | `step` | OCR 结果标准化。 |
| `translation` | `translation_prepare` | `step` | 翻译准备。 |
| `translation` | `domain_inference` | `step` | 文档领域识别。 |
| `translation` | `page_policies` | `page` | 页面策略和块分类。 |
| `translation` | `continuation_review` | `page` | 跨栏/跨页连续段复核。 |
| `translation` | `translation_batches` | `batch` | 主翻译批次。 |
| `translation` | `translation_tail_retry` | `batch` | 尾部重试队列。 |
| `translation` | `garbled_repair` | `page` | 乱码候选段修复。 |
| `translation` | `agent_repair` | `page` | 翻译结果修复。 |
| `translation` | `final_untranslated_recovery` | `page` | 最终未翻译收口。 |
| `render` | `render_prepare` | `step` | 渲染准备。 |
| `render` | `render_prewarm` | `step` | 渲染预热；通常是 `lane=background`。 |
| `render` | `render_pages` | `page` | 页面构建、清理、覆盖、Typst source 生成。 |
| `render` | `render_compile` | `step` | Typst 编译、PDF 保存等不可按页拆分步骤。 |

进度规则：

- 同一个 `stage + substage + progress.unit` 下，`progress.current` 不应倒退。
- `render_compile` 这类 step 进度不要显示成“共 N 页”。
- `render_prewarm` 属于后台 lane，不应覆盖 `render_pages` 的主进度。
- 完成态事件应尽量带最终 `current=total`，方便刷新后恢复状态。

## 生命周期

常见 workflow：

- `book`
- `ocr`
- `translate`
- `render`

常见主阶段：

- `queued`
- `ocr_submitting`
- `ocr_processing`
- `normalizing`
- `translation_prepare`
- `translating`
- `render_prepare`
- `render_preprocess`
- `rendering`
- `saving`
- `finished`
- `failed`

## 失败协议

失败时优先看：

- `failure.summary`
- `failure.category`
- `failure.root_cause`
- `failure.retryable`
- `failure.suggestion`

兼容字段：

- `failure_diagnostic`
- `log_tail`

前端如果只需要任务详情页的干净失败信息，优先读：

`GET /api/v1/jobs/{job_id}/diagnostics`

返回字段：

- `failed_stage`
- `failed_substage`
- `summary`
- `detail`
- `suggestion`
- `retryable`
- `resume_available`

这个接口会复用 `failure` 的正式字段，并在必要时回退到后端分类器，不要求前端自己解析 `request_payload`、`events` 或 `log_tail`。

## 断点恢复

恢复计划：

`GET /api/v1/jobs/{job_id}/resume-plan`

返回字段：

- `can_resume`
- `job_id`
- `from_stage`
- `resume_workflow`
- `reuses_artifacts`
- `reruns_stages`
- `reason`

当前规则：

- 有 `translations_dir + source_pdf`：`from_stage=render`，`resume_workflow=render`，只重跑渲染。
- 有 `normalized_document_json + source_pdf`：`from_stage=translate`，`resume_workflow=book`，重跑翻译和渲染。
- 两类 checkpoint 都没有：`can_resume=false`，`reason` 给出原因。

执行恢复：

`POST /api/v1/jobs/{job_id}/resume`

当前实现复用 `/rerun` 的执行契约：

- 可原地重渲染时，保留原 `job_id`，清空旧渲染产物，排队 `render` workflow。
- 只有 OCR checkpoint 时，创建新的 `book` 恢复任务。
- 如果任务仍在 `queued` / `running`，会返回状态冲突，需要先取消。

## 关键结论

- 详情页主接口是 `GET /api/v1/jobs/{job_id}`
- 时间线看 `runtime.stage_history`
- 事件流看 `GET /api/v1/jobs/{job_id}/events`
- 失败真源看 `failure`
- 简洁失败信息看 `GET /api/v1/jobs/{job_id}/diagnostics`
- 恢复按钮状态看 `GET /api/v1/jobs/{job_id}/resume-plan`
