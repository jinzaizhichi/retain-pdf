# 阶段事件与失败协议

这份文档定义 Rust API 和 Python worker 之间的阶段事件、进度事件和失败对象边界。

目标是让前端不再靠 `message`、`log_tail` 或 stdout 文本猜状态。

## 1. 三层事件来源

当前后端存在三层事件：

1. Python 原始 pipeline events
   - 文件：`DATA_ROOT/jobs/{job_id}/logs/pipeline_events.jsonl`
   - 写入方：Python OCR / translation / render worker
   - 用途：阶段内细粒度进度、artifact 发布、失败上下文

2. Rust DB events
   - 表：SQLite `events`
   - 写入方：Rust job runner
   - 用途：job 创建、状态变更、终态、兼容旧事件

3. Rust API canonical events
   - 接口：`GET /api/v1/jobs/{job_id}/events`
   - 组装方：`services/jobs/presentation/live_stage`
   - 用途：前端和第三方唯一应消费的公共事件流

前端只消费第三层。第一层和第二层是后端内部来源。

## 2. Python 原始事件契约

Python 写入 `pipeline_events.jsonl` 时，必须稳定带这些字段：

```json
{
  "job_id": "job-123",
  "seq": 12,
  "ts": "2026-05-28T10:12:33Z",
  "created_at": "2026-05-28T10:12:33Z",
  "level": "info",
  "user_stage": "translation",
  "stage": "translating",
  "substage": "translation_batches",
  "stage_detail": "已完成第 18/55 批翻译",
  "event_type": "stage_progress",
  "semantic_event_type": "progress",
  "message": "已完成第 18/55 批翻译",
  "progress_current": 18,
  "progress_total": 55,
  "progress_unit": "batch",
  "provider": "",
  "provider_stage": "",
  "elapsed_ms": 193822,
  "payload": {}
}
```

字段规则：

- `user_stage`
  Python 侧粗阶段，只允许：`ocr | translation | render | done`。
- `stage`
  Python 内部机器阶段，例如 `translating`、`render_preprocess`。
- `substage`
  机器可读子阶段，是前端状态语义的核心字段。
- `stage_detail`
  给用户看的短文案。
- `message`
  日志文案，只给人看；不要承载机器语义。
- `progress_current` / `progress_total` / `progress_unit`
  必须和当前 `user_stage + substage` 对齐。
- `semantic_event_type`
  Python 对旧 `stage_progress` / `artifact_published` 等事件的语义归一化，Rust 仍会二次 canonicalize。
- `payload`
  只能放扩展数据，不放主语义。

禁止出现这种事件：

```json
{
  "user_stage": "translation",
  "stage": "translating",
  "message": "render payload prewarm: ready"
}
```

正确写法：

```json
{
  "user_stage": "render",
  "stage": "render_preprocess",
  "substage": "render_prewarm",
  "message": "render payload prewarm: ready"
}
```

## 4. Rust canonical event 契约

Rust API 对外返回的事件不是原始 Python 行，而是 canonical event。

对外字段示例：

```json
{
  "job_id": "job-123",
  "seq": 24,
  "ts": "2026-05-28T10:12:33Z",
  "created_at": "2026-05-28T10:12:33Z",
  "level": "info",
  "lane": "main",
  "stage": "translation",
  "substage": "translation_batches",
  "stage_detail": "已完成第 18/55 批翻译",
  "event": "stage_progress",
  "event_type": "progress",
  "raw_event_type": "stage_progress",
  "progress": {
    "unit": "batch",
    "current": 18,
    "total": 55,
    "percent": 32.72727272727273
  },
  "message": "已完成第 18/55 批翻译",
  "payload": {
    "raw_stage": "translating",
    "raw_user_stage": "translation",
    "raw_event_type": "stage_progress"
  }
}
```

公共 `stage` 只表达前端显示层：

- `ocr`
- `translation`
- `render`
- `done`

公共 `event_type` 只表达事件类型：

- `progress`
- `artifact`
- `terminal`
- `error`
- `diagnostic`

`user_stage` 是内部来源字段，API 输出会默认隐藏。前端不要依赖它。

## 5. lane 规则

`lane` 用来解决“后台事件覆盖主状态”的问题。

- `main`
  主进度。状态卡可以显示。
- `background`
  后台预热、缓存构建等。不能覆盖主阶段。
- `artifact`
  产物发布。不能覆盖主阶段。
- `diagnostic`
  诊断和错误辅助信息。

当前固定规则：

- `substage=render_prewarm` -> `lane=background`
- `event_type=artifact` -> `lane=artifact`
- `event_type=diagnostic` 或失败辅助事件 -> `lane=diagnostic`
- 其他进度事件 -> `lane=main`

## 6. live stage 快照

`GET /api/v1/jobs/{job_id}` 和列表接口会从 canonical events 中选当前 live stage。

选择规则：

- 只从 `lane=main` 中选主阶段。
- `artifact` 和 `background` 事件不能覆盖主阶段。
- render 阶段优先展示 `render_pages` 的页级进度；`render_compile` 是 step 进度，不能伪装成“共 N 页”。
- 失败时如果终态事件没有进度，会保留最后可信进度。

详情接口中的 `progress` 是当前快照：

```json
{
  "current": 120,
  "total": 533,
  "percent": 22.5140712945591,
  "unit": "page"
}
```

前端刷新页面时优先读详情接口的 `stage`、`stage_detail`、`progress`，事件流用于补充实时变化。

## 7. 进度单调性

Python writer 会按 `(user_stage, substage, progress_unit)` 维护单调进度。

要求：

- 同一 key 下 `progress_current` 不倒退。
- 不同 substage 可以独立计数，例如 `render_pages` 和 `render_compile` 互不影响。
- `render_prewarm` 是后台 step，不参与主渲染页数。
- 新任务或重试任务必须重新写事件，不复用旧任务事件 seq。

## 8. 失败协议

正式失败对象是 job detail 中的 `failure`。

示例：

```json
{
  "failed_stage": "translation",
  "provider": "translation",
  "provider_stage": "continuation_review",
  "failure_code": "upstream_timeout",
  "failure_category": "timeout",
  "provider_code": null,
  "summary": "翻译阶段超时",
  "root_cause": "provider timed out",
  "retryable": true,
  "upstream_host": "api.deepseek.com",
  "suggestion": "从断点恢复任务",
  "last_log_line": "request timed out",
  "raw_excerpt": "timeout"
}
```

字段分层：

- `failed_stage`
  公共失败阶段，优先使用 `ocr | translation | render | done`。
- `provider` / `provider_stage`
  上游或内部子阶段归属。
- `failure_code`
  稳定机器码。
- `failure_category`
  粗粒度错误类别。
- `provider_code`
  上游 provider 原始错误码。
- `summary` / `root_cause` / `suggestion`
  面向前端和操作者的可读信息。
- `raw_excerpt`
  脱敏后的原始片段。

`failure_diagnostic` 是兼容投影，不是新代码的写入目标。

前端需要干净失败信息时，优先用：

`GET /api/v1/jobs/{job_id}/diagnostics`

## 9. 验收标准

事件和失败协议改动必须满足：

- 前端不靠 `message` 判断阶段。
- render 相关事件不会带 `user_stage=translation`。
- `render_prewarm` 不覆盖主渲染进度。
- `/events` 输出只暴露公共 `stage` 和稳定 `substage`。
- `/jobs/{job_id}` 的 `progress` 是刷新后可用的最终快照。
- 失败信息优先来自 `failure`，不是 `log_tail` 或 stdout 文本。
