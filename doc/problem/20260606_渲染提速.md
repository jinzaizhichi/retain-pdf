# 20260606 渲染提速：PDF 物理删除底层优化记录

## 背景

RetainPDF 的渲染阶段目前最核心的耗时点不是 Typst 文本绘制本身，而是源 PDF 的清理与预处理，尤其是“物理删除原文文字”这条链路。

对于普通可编辑 PDF，理想路径是：

1. 根据 OCR / normalized document 的 bbox 找到需要删除的原文区域。
2. 用 pikepdf 解析并改写页面 content stream。
3. 删除命中区域内的文字绘制 token。
4. 再由 Typst 绘制译文。

这条路径的价值是输出更干净，不依赖白块覆盖，也能减少浏览器 PDF 渲染里背景文字残留的问题。但在大 PDF、Form XObject 多、矢量路径多、页面对象复杂的场景下，旧实现会非常慢。

本次优化目标不是继续补规则，而是从底层降低算法复杂度和重复 IO，把 500 页级别 PDF 的物理删除耗时压到可接受范围内。当前阶段的目标是：大部分正常 PDF 在几十秒内完成源 PDF 清理，复杂页走明确 fallback，不把所有页面都拖慢。

## 原始瓶颈

### 1. 页面几何扫描太重

旧逻辑里为了判断某个 bbox 附近有没有图片、线条、公式或复杂对象，经常会调用 PyMuPDF 的 `page.get_bboxlog()`。这个接口会展开页面绘制日志。

问题是：

- 普通页面还可以接受。
- Form XObject 页面会被展开成内部对象。
- 矢量图、图表、复杂背景可能产生大量 path / image / fill / stroke 记录。
- planning 阶段其实只需要“是否危险”和“哪些区域可能冲突”，不一定需要展开所有底层对象。

所以旧逻辑在复杂页上会把“判断是否要删文字”变成一次昂贵的页面渲染级扫描。

### 2. `item x rect` 全量扫描

很多判断本质上是矩形相交问题：

- OCR item bbox 是否覆盖 unsafe vector rect。
- target rect 是否碰到 drawing rect。
- 删除区域是否需要避开 guard rect。

旧实现经常是两层循环：

```text
for item in items:
    for rect in unsafe_rects:
        if overlap(item, rect):
            ...
```

如果页面上有几百个 OCR item、几千个绘制 rect，这类 `O(n*m)` 扫描会快速放大。大 PDF 的页数再乘上去，整体耗时就不稳定。

### 3. 全局 rect merge 接近 `O(n^2)`

删除候选矩形需要合并，避免生成太碎的 strip rect。旧的合并方式容易在所有 rect 之间反复比较，最坏情况下接近 `O(n^2)`。

这类问题在 TOC、参考文献、表格、公式密集页面上会被放大。

### 4. rewrite worker 负载不均

pikepdf 解析和改写 content stream 是 CPU-bound 的。旧逻辑如果按页码顺序切 chunk，可能出现某个 worker 拿到一批特别重的页面，其他 worker 已经结束，整体还在等最后一个重 chunk。

典型现象是：总页数不算夸张，但最后一段 rewrite 拖很久。

### 5. prewarm 与 render 重复计算稳定事实

页面是否含 Form XObject、content stream 大小、删除候选页等信息，在同一个 job 里基本是稳定的。旧实现中 planning、prewarm、render 之间存在重复扫描和重复判断，导致相同成本被付了多次。

## 当前底层优化方案

### 执行链路

当前物理删除主链路可以拆成四层：

```text
normalized document / translated items
    ↓
planning: 生成每页 strip rect / protected rect / skip metadata
    ↓
manifest: 缓存 page features 与候选删除计划
    ↓
rewrite: pikepdf 解析页面 content stream 并删除文字 show op
    ↓
render source: 根据 changed / skipped / no-effect 决定是否进入 fallback
```

对应文件大致是：

```text
source_cleanup/planning/planner.py
    负责把 OCR / 译文 item 转成页面级删除计划。

source/prewarm_manifest_io.py
source/prewarm_contracts.py
    负责把删除候选、页面特征、skip 结果持久化。

source_cleanup/pdf/document.py
    负责打开 PDF、并发改写页面 content stream、保存输出。

source_cleanup/pdf/stream_engine.py
    负责真正解析 content stream token 并判断 Tj/TJ 等文字绘制是否命中删除区域。

source/render_source.py
    负责把物理删除结果接到后续渲染源选择与 fallback。
```

需要注意的是，planning 只应该回答“哪些页面、哪些区域值得尝试删除”。rewrite 才真正改 PDF。fallback 不应该反向影响 planning 的底层事实判断，否则后面会继续出现规则打架。

### 1. 用空间索引替代全量矩形扫描

核心文件：

- `backend/scripts/services/rendering/source_cleanup/planning/spatial_index.py`

新增 `RectOverlapIndex`，把页面上的 rect 按 y 方向排序，然后用 `bisect_right` 找到可能相交的候选范围，再做 x 方向 quick reject。

它解决的是这类问题：

```text
item bbox -> 是否和 unsafe vector rect 相交
target rect -> 是否和 drawing rect 相交
```

当前实现非常轻量：

```text
RectOverlapIndex
    rects: 按 y0 排序后的矩形
    y0_sorted: 单独抽出的 y0 数组，用于 bisect

overlaps_any(target_rect)
    1. 用 target_rect.y1 在 y0_sorted 上二分，拿到可能相交的前缀。
    2. 跳过 y1 < target.y0 的矩形。
    3. 用 x0/x1 做快速排除。
    4. 最后才计算真实相交面积。
```

旧逻辑是每个 item 扫所有 rect。新逻辑先通过 y 区间剪枝，只检查少量可能相交的 rect。

底层复杂度从接近：

```text
O(items * rects)
```

降为更接近：

```text
O(rects log rects + items * local_candidates)
```

这里的 `local_candidates` 通常远小于全页 rect 数量。

### 2. 页面坐标与 bboxlog 一次分组

核心文件：

- `backend/scripts/services/rendering/source_cleanup/planning/coordinate_resolver.py`

旧逻辑容易在不同判断里重复读取同一页的几何信息。现在把页面坐标、text/image/vector rect 的提取收口成一次分组。

它的意义是：

- 同一页只做一次昂贵的 bboxlog 解析。
- text rect、image rect、unsafe vector rect 在同一个结果里返回。
- 后续 planner 只消费结构化结果，不再到处自己扫页面。
- 顺便选择 OCR bbox 的坐标解释方式，减少“向左/向上偏移”这类坐标系错误。

这不是单纯为了代码整洁，而是避免每个策略模块各自调用一次底层 PDF 扫描。

这里的关键结构是：

```text
PageBBoxResolver
    text_rects
    text_index
    image_rects
    unsafe_vector_rects
    unsafe_vector_index
    preferred_candidate
```

`preferred_candidate` 会用 OCR bbox 和页面 text rect 的重叠分数来选择坐标解释方式。这样比在业务层到处猜 `top_left` / `pdf_matrix` 更稳定。

### 3. Form XObject 页面走 fast plan，不展开内部几何

核心文件：

- `backend/scripts/services/rendering/source_cleanup/planning/page_probe.py`
- `backend/scripts/services/rendering/source_cleanup/planning/planner.py`

Form XObject 是之前最容易拖慢的点。因为 MuPDF 会把 XObject 内部内容展开，而 pikepdf 的页面级 content stream 改写并不能可靠删除 XObject 内部文字。

所以新策略是：

1. 用便宜的 `page.get_xobjects()` 或页面特征判断是否含 Form XObject。
2. planning 阶段不展开 XObject 内部 bboxlog。
3. 这类页只基于 OCR bbox 生成页面级候选。
4. execution 阶段保留 `skipped_form_xobject` 元数据。
5. 如果物理删除对这类页无效，交给后续 cover fallback 兜底。

这个设计的关键是承认边界：页面级 pikepdf 改写不是万能的。Form 内部文字删不掉时，不应该为了“尝试删除”而把全书 planning 拖慢。

这不是放弃删除，而是把成本和语义分开：

```text
页面级文字流能删 -> pikepdf strip
Form 内部文字不确定 -> 标记 skipped_form_xobject
实际仍有残留 -> cover / visual background fallback
```

这样后续排查时不会把“计划阶段跳过复杂内部展开”和“最终没有覆盖残留英文”混在一起。

### 4. 页面特征缓存进 prewarm manifest

核心文件：

- `backend/scripts/services/rendering/source_cleanup/planning/page_features.py`
- `backend/scripts/services/rendering/source/prewarm_manifest_io.py`
- `backend/scripts/services/rendering/source/prewarm_contracts.py`

新增 `PageCleanupFeatures`，记录：

- `content_stream_size`
- `has_form_xobjects`

这些是后续 planning、调度、fallback 都会用到的稳定事实。现在会随着 prewarm manifest 读写，避免 render 阶段重新探测。

manifest 版本已升级到：

```text
bbox_text_strip_v18_page_features_lpt
```

### 5. y-band rect merge，避免全局两两比较

核心文件：

- `backend/scripts/services/rendering/source_cleanup/planning/rects.py`

rect merge 从全局暴力比较改成按 y-band 维护 active window。

直觉上，两个矩形如果 y 方向距离已经很远，就不可能合并，也不需要比较。新实现利用这个事实，只在局部 y 窗口里尝试 merge。

这可以明显降低 TOC、参考文献、公式密集页里的合并成本。

### 6. vector text cleanup 也接入空间索引

核心文件：

- `backend/scripts/services/rendering/source/vector_text.py`

vector text cleanup 原来同样存在 `drawing x target_rects` 的扫描。现在 target rect 也建空间索引，drawing 只查询可能相交的目标区域。

这个优化和 source cleanup 的思路一致：所有矩形相交问题都不应该默认写成双层全扫。

### 7. pikepdf rewrite 使用 LPT 调度

核心文件：

- `backend/scripts/services/rendering/source_cleanup/pdf/document.py`

pikepdf 改写阶段用 decoded content stream size 估算页面重量，然后采用 LPT（Longest Processing Time first）分配 chunk。

做法：

1. 为每页估算 rewrite weight。
2. 按 weight 从大到小排序。
3. 每次把最重页面分给当前累计 weight 最小的 worker。

这样可以减少“最后一个 worker 拿到最重页面”的长尾问题。

目前 weight 的来源是 decoded content stream size：

```text
_page_content_stream_weight(pdf, page_idx)
    /Contents 是单 stream -> len(read_bytes())
    /Contents 是 array -> 多个 stream 长度求和
    读取失败 -> 退回 /Length
```

这个权重不完美，但它足够便宜，并且和 pikepdf parse/unparse 成本通常正相关。

日志里会输出 `top_chunks`，方便观察是否仍然存在极端重页：

```text
top_chunks=34p/3289283b/11.41s,...
```

### 8. runtime skip 元数据保留

相关结构：

- `BBoxTextStripCandidates`

现在会保留：

- `skipped_form_xobject`
- `strip_no_effect`
- `page_features`

这些信息会进入 manifest round-trip。后续渲染、事件、fallback 可以明确知道：

- 哪些页是因为 Form XObject 没做精确删除。
- 哪些页尝试过物理删除但没有效果。
- 哪些页本来就不该走重型扫描。

这比单纯返回“删除成功/失败”更可靠，也方便后续做策略分层。

## 为什么这些优化不会改变核心删除语义

本轮优化主要改的是“找候选”和“调度执行”，不是改文字 token 删除规则。

不改变语义的部分：

- `RectOverlapIndex` 只减少候选相交检查数量，最终仍然用真实 rect intersection 判断。
- y-band merge 只跳过 y 方向已经不可能合并的矩形，局部窗口内仍然使用原来的 `rects_should_merge`。
- LPT 只改变并发 worker 分配顺序，最终结果仍按 page index 写回。
- `PageCleanupFeatures` 是缓存稳定页面事实，不直接决定删哪些 token。

有意改变策略边界的部分：

- Form XObject 页不再在 planning 阶段展开内部几何。
- 这类页通过 `skipped_form_xobject` 明确暴露给 fallback。

这个改变是为了避免昂贵且语义不可靠的内部展开。因为页面级 content stream 中只有 `Do` 调用时，继续展开 bboxlog 并不会让 pikepdf 页面级删除天然变得可靠。

## 如何定位下一次慢在哪里

先看后端日志里的这一行：

```text
bbox text strip: mode=strip pages=... text_show_ops=... candidates=... open=... rewrite=... apply=... make_stream=... assign=... save=... top_apply_pages=... top_chunks=...
```

字段含义：

- `candidates`：planning 候选生成耗时。
- `open`：pikepdf 打开源 PDF 的耗时。
- `rewrite`：content stream parse / strip 的耗时。
- `apply`：把改写后的 stream 写回 pikepdf 对象的耗时。
- `make_stream`：创建新 PDF stream 的耗时。
- `assign`：替换页面 `/Contents` 的耗时。
- `save`：保存输出 PDF 的耗时。
- `top_apply_pages`：写回阶段最慢的页面。
- `top_chunks`：rewrite worker 里最慢的 chunk。

判断方式：

```text
candidates 慢
    优先查 planning 是否又引入了重复 bboxlog 或全量 rect scan。

rewrite 慢
    优先查 content stream 特别大的页面、worker chunk 是否不均、是否有重复 parse。

save 慢
    优先查输出 PDF 对象数量、压缩参数、是否反复生成中间文件。

top_chunks 明显只有一个 chunk 特别慢
    优先查 LPT 权重是否低估了某类页面。

skipped_form_xobject 很高且最终有英文残留
    优先查 fallback 是否覆盖了 skipped / no-effect 页，而不是继续加物理删除规则。
```

## 设计原则

### 不再用 job/page 规则修补

删除不干净的问题不能靠不断加：

```text
如果是第 x 页
如果 bbox 很像公式
如果某个 block id
```

这类规则会互相打架，也会拖慢大 PDF。现在的方向是把 PDF 页面拆成稳定的底层事实：

- 页面是否有 Form XObject。
- 页面 content stream 大小。
- 页面哪些 OCR bbox 需要删除。
- 哪些候选区域物理删除无效。
- 哪些页应该直接进入 cover fallback。

策略只消费这些事实，不直接依赖具体 job 特例。

### 对复杂页承认边界

pikepdf 页面级 content stream 改写适合删除页面流里的文字。但如果文字藏在 Form XObject 内部，页面流里只有一个 `Do` 调用，强行在页面级删文字并不可靠。

因此复杂页要尽早识别，避免重型展开。删不掉的页保留明确元数据，再由 cover fallback 或背景渲染兜底。

### 所有矩形问题优先索引化

渲染清理里大量逻辑本质都是 rect overlap。后续新增功能时，默认不应该写 `A x B` 双层全扫，而应该先问：

- rect 数量是否可能增长？
- 是否可以按页建索引？
- 是否可以按 y-band / x-range 剪枝？
- 是否可以在 prewarm 阶段缓存？

## 后续改代码时的边界

### planning 层

应该放：

- bbox 坐标转换。
- 页面级特征探测。
- strip rect / protected rect 生成。
- 跳过原因和候选元数据。
- 快速、可缓存、可复用的矩形判断。

不应该放：

- 真实 content stream token 删除。
- 大量 PDF 对象重写。
- 根据单个 job/page 的特殊规则。
- 和前端事件展示耦合的逻辑。

### pdf rewrite 层

应该放：

- pikepdf 打开、解析、改写、保存。
- 并发调度。
- content stream 权重估算。
- 删除 token 数量、改动页数、耗时统计。

不应该放：

- OCR item 语义判断。
- 翻译状态判断。
- “这段是不是正文/公式/标题”的业务分类。

### render source / fallback 层

应该放：

- 根据物理删除结果选择源 PDF。
- 对 `skipped_form_xobject` / `strip_no_effect` 页启用 cover 或 visual background。
- 输出最终给 Typst / PDF 合成使用的背景源。

不应该放：

- 重新扫描全书生成删除计划。
- 修改 planner 的底层候选逻辑。

这个边界很重要。否则后面又会变成“渲染发现残留英文 -> fallback 改 planner -> planner 又影响物理删除速度”的循环。

## 后续还能继续优化的方向

### 1. rewrite 层继续减少 parse/unparse 次数

目前主要耗时已经在 pikepdf rewrite。后续可以继续检查：

- 是否有页面被重复 parse。
- 是否能按 object stream 缓存 decoded content。
- 是否可以只 rewrite 有命中 token 的 stream。
- 是否能把多个小 stream 合并处理，减少 Python 层调度开销。

### 2. 把更多稳定事实前移到 OCR 后预处理

OCR 完成后就已经知道 source PDF、normalized document 和大部分 bbox。可以在 OCR 后立即做：

- page feature probe
- candidate planning
- background / pseudo-copyable 判断
- 删除候选 manifest 生成

这样翻译进行时，渲染预处理可以同步完成，render 阶段只消费现成 manifest。

### 3. 对 Form/XObject 页做更明确的策略分流

当前策略是 fast plan + skip metadata + fallback。后续可以进一步做：

- Form 内文字占比高的页直接 visual-only background。
- 普通页面继续 pikepdf physical strip。
- 混合页面只对页面级文本做 strip，Form 区域强制 cover。

关键是分流要依赖页面特征，不依赖手写 job 规则。

### 4. 给事件和日志增加性能字段

为了后续定位慢页，建议持续输出：

- planning elapsed
- rewrite elapsed
- save elapsed
- candidate page count
- changed page count
- skipped form count
- no-effect count
- top heavy chunks

这样前端看到“渲染慢”时，后端可以直接判断是 planning 慢、rewrite 慢、save 慢，还是 fallback 触发太多。

## 当前结论

本轮提速的本质是三件事：

1. **减少昂贵页面扫描**：复杂页不展开，稳定页面事实缓存进 manifest。
2. **降低算法复杂度**：矩形相交用空间索引，rect merge 用 y-band。
3. **减少 rewrite 长尾**：pikepdf 改写按页面重量做 LPT 调度。

这使得 500 页级别样本的物理删除主链路从接近半分钟进一步降到约 13 秒，planning 本身降到约 0.6 秒。后续如果要继续提升，重点已经不在规则层，而在 pikepdf content stream rewrite 的 parse/unparse 次数、页面分流，以及 OCR 后预处理并行化。
