# 任务详情与时间线

## 1. 主接口

任务详情页的主接口是：

`GET /api/v1/jobs/{job_id}`

详情页的大部分核心信息都应从这一个接口拿：

- 当前状态
- 当前阶段
- 当前阶段说明
- 运行态时间线
- 失败归因
- 下载按钮状态
- OCR 子任务摘要

## 2. 前端最该关心的字段

重点字段在 `data` 里：

- `request_payload`
- `status`
- `stage`
- `stage_detail`
- `runtime`
- `failure`
- `actions`
- `artifacts`
- `timestamps`
- `log_tail`

其中：

- `request_payload` 是后端为该任务实际保存的请求参数快照
- 联调时如果要确认某个参数到底有没有进后端，优先看这里
- 例如 OCR 页码范围应读：`data.request_payload.ocr.page_ranges`

## 3. 时间线真源

“概览 -> 过程时间线”必须以 `runtime.stage_history` 为准。

不要用 `/events` 反推主时间线。原因很直接：

- `runtime.stage_history` 已经是后端整理好的阶段段落
- 每段都有进出时间
- 每段可以直接挂终态信息
- 前端不需要再做合并、去重、推断

## 4. `runtime` 里有哪些关键字段

当前重点字段：

- `current_stage`
- `stage_started_at`
- `last_stage_transition_at`
- `total_elapsed_ms`
- `active_stage_elapsed_ms`
- `retry_count`
- `last_retry_at`
- `terminal_reason`
- `final_failure_category`
- `final_failure_summary`
- `stage_history`

## 5. `stage_history` 的固定结构

每一项都是一个阶段时间片段：

```json
{
  "stage": "translating",
  "detail": "正在翻译，第 12/22 批",
  "enter_at": "2026-04-04T15:31:02Z",
  "exit_at": null,
  "duration_ms": null,
  "terminal_status": null
}
```

字段含义：

- `stage`：阶段名
- `detail`：进入该阶段时的主要说明
- `enter_at`：进入该阶段的时间
- `exit_at`：离开该阶段的时间；当前活跃阶段通常为空
- `duration_ms`：阶段已完成时才稳定；当前活跃阶段通常为空
- `terminal_status`：若该阶段就是结束前最后阶段，可标记 `succeeded / failed / canceled`

## 6. 运行中任务怎么读

如果任务还在运行：

- 当前活跃阶段也会出现在 `stage_history`
- 该项通常 `exit_at = null`
- 该项通常 `duration_ms = null`
- 当前阶段实时耗时应读 `runtime.active_stage_elapsed_ms`

也就是说，前端不要自己用“当前时间 - enter_at”硬算展示值，除非只是本地补帧；接口真源仍应是 `active_stage_elapsed_ms`。

## 7. 已结束任务怎么读

如果任务已结束：

- `status` 会是 `succeeded / failed / canceled`
- `runtime.terminal_reason` 会说明终态原因
- `runtime.total_elapsed_ms` 是整条链路总耗时
- `runtime.stage_history` 最后一项通常会带 `terminal_status`

## 8. 历史任务为什么可能没有时间线

这是最近最容易误判的地方。

历史老任务可能出现：

```json
{
  "runtime": null
}
```

这不表示：

- 前端读错了
- 当前后端坏了
- 当前任务写入失败了

它只表示：

- 这条任务创建和执行时，后端还没有把运行态时间线持久化到数据库

所以这类任务前端应视为“历史数据缺失”。

## 9. 保证范围

当前文档里的保证范围是：

- 任务由新版后端创建
- 任务由同一套新版后端全程执行

只有在这个范围内，`runtime.stage_history` 才保证是完整全过程，而不是只保留最后阶段。

## 10. 适合前端直接使用的读取顺序

1. 先读 `status`
2. 再读 `runtime.current_stage`
3. 时间线直接读 `runtime.stage_history`
4. 当前阶段耗时读 `runtime.active_stage_elapsed_ms`
5. 总耗时读 `runtime.total_elapsed_ms`
6. 失败任务优先读 `failure.summary` 和 `failure.category`
