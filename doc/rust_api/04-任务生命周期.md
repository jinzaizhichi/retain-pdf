# 任务生命周期

## 1. 主链路

当前主任务大体是这条链路：

1. 上传 PDF
2. 创建主任务
3. 主任务内部创建 OCR 子任务
4. OCR provider 上传、轮询、下载结果
5. 标准化为 `document.v1`
6. 翻译
7. 渲染
8. 输出 PDF / ZIP / 其他产物

## 2. 为什么会有 OCR 子任务

主任务不直接把所有事情堆在一起做，而是会产生一个 OCR 子任务，通常是：

`{job_id}-ocr`

这样做有两个好处：

- OCR transport 和主翻译渲染链路可以分开观察
- OCR 的 provider 诊断可以单独挂在详情里返回

## 3. 常见阶段名

阶段名会随着链路推进变化，常见包括：

- `queued`
- `ocr_submitting`
- `ocr_upload`
- `mineru_upload`
- `mineru_processing`
- `translation_prepare`
- `normalizing`
- `translating`
- `domain_inference`
- `page_policies`
- `rendering`
- `finished`
- `failed`

不是每条任务都会经过完全相同的阶段，但主思路一致。

## 4. 前端怎么理解阶段

建议区分三层：

- `status`：任务是否结束
- `stage`：当前在哪个阶段
- `stage_detail`：当前阶段的人类说明

例如：

- `status = running`
- `stage = translating`
- `stage_detail = 已完成第 18/55 批翻译`

这说明：

- 任务还没结束
- 当前阶段是翻译
- 当前批次进度已经走到 18/55

## 5. 下载按钮什么时候可点

前端不要自己拼规则，而要直接看详情接口里的：

- `actions.*.enabled`
- `artifacts.*.ready`

例如：

- PDF 下载按钮看 `actions.download_pdf.enabled`
- ZIP 下载按钮看 `actions.download_bundle.enabled`

## 6. 失败时应该看什么

失败任务优先看：

- `failure.category`
- `failure.summary`
- `failure.root_cause`
- `failure.retryable`
- `failure.suggestion`

如果还不够，再看：

- `runtime.final_failure_category`
- `runtime.final_failure_summary`
- `log_tail`
- `/events`

## 7. 时间线和事件流的职责分工

这个分工非常重要：

- `runtime.stage_history` 回答“全过程每个阶段花了多久”
- `/events` 回答“过程中发生过哪些具体事件”

前者适合概览，后者适合排错。
